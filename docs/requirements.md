# 要件定義書 — 個人技術ブログアプリ

> 作成日: 2026-03-18
> ステータス: ドラフト v1.0

---

## 1. プロジェクト概要

### 1.1 目的

技術系の記事を自由に記録・公開するための個人ブログアプリを構築する。
Zenn などのプラットフォームのガイドラインに縛られず、自分のペースでアウトプットを蓄積できる場所を持つことを目的とする。

### 1.2 背景

- 現在 Notion データベース + Notion API を使った自作ブログアプリが存在するが、コード品質・デザイン・機能に課題がある
- ゼロから再設計し、保守性・使いやすさ・見た目を向上させる

### 1.3 ターゲットユーザー

- **主な利用者**: 筆者本人（記事閲覧・管理）
- **副次的な利用者**: ブログを読みに来る不特定多数の読者

## 2. 機能要件

### 2.1 必須機能（Must）

| # | 機能 | 説明 |
|---|------|------|
| F-01 | 記事一覧表示 | 公開済み記事をカード形式で一覧表示する |
| F-02 | 記事詳細表示 | 記事本文（Markdown/リッチテキスト）を整形して表示する |
| F-03 | タグ・カテゴリ分類 | 記事にタグとカテゴリを付与し、絞り込み表示できる |
| F-04 | 記事検索 | キーワードで記事タイトル・本文を検索できる |
| F-05 | コードシンタックスハイライト | コードブロックを言語ごとにカラーリングして表示する |

### 2.2 推奨機能（Should）

| # | 機能 | 説明 |
|---|------|------|
| F-06 | OGP 対応 | SNS シェア時にタイトル・説明・OGP 画像を表示する |
| F-07 | SEO メタデータ | 各記事に `<title>` / `<meta description>` を設定する |
| F-08 | ページネーション / 無限スクロール | 記事が増えても一覧が重くならないようにする |
| F-09 | 目次（TOC）自動生成 | 長い記事の見出しから目次を自動生成して表示する |

### 2.3 将来対応（Could）

| # | 機能 | 説明 |
|---|------|------|
| F-10 | RSS フィード | `/feed.xml` で記事をフィード配信する |
| F-11 | ダークモード | ユーザーのシステム設定に応じて配色を切り替える |
| F-12 | 関連記事表示 | タグ・カテゴリをもとに関連記事をサジェストする |

---

## 3. 非機能要件

| カテゴリ | 要件 |
|----------|------|
| パフォーマンス | Lighthouse Score 90 以上（Performance / SEO / Accessibility） |
| レスポンシブ | スマートフォン・タブレット・PC すべてで正常に表示される |
| アクセシビリティ | 見出し構造・alt テキスト・キーボード操作の基本対応 |
| コード品質 | TypeScript 必須・ESLint / Prettier 設定・コンポーネント単位で責務を分離 |
| セキュリティ | API キーは環境変数で管理し、クライアントに露出しない |

## 4. 技術スタック

### 4.1 確定事項

| レイヤー | 技術 | 理由 |
|----------|------|------|
| フロントエンド | **Next.js 15 (App Router)** | SSG/ISR でブログに最適・TypeScript との相性◎ |
| 言語 | **TypeScript** | 型安全でコード品質を担保 |
| スタイリング | **Tailwind CSS** | ユーティリティファーストで素早く高品質な UI を構築 |
| シンタックスハイライト | **Shiki** または **Prism.js** | コードブロックの見栄えを向上 |
| デプロイ | **Vercel** または **Cloudflare Pages** | Next.js と相性が良い・無料枠で運用可能 |

### 4.2 要検討事項 — コンテンツ管理バックエンド

以下の 3 案から選択する。それぞれにトレードオフがある。

#### 案 A: Notion API（現行踏襲）

```
Notion DB → Notion API → Next.js（ISR/SSG）
```

| 利点 | 欠点 |
|------|------|
| Notion で快適に執筆できる | API レート制限がある |
| リッチな編集 UI がすでにある | Notion のブロック構造をパースする処理が複雑 |
| 現行の書き方を変えなくてよい | Notion に依存（サービス障害・料金変更リスク） |

#### 案 B: Markdown ファイル（Git 管理）✅ **採用**

```
.md ファイル → git push → GitHub Actions → Vercel/Cloudflare 自動デプロイ
```

| 利点 | 備考 |
|------|------|
| シンプルでコード品質を保ちやすい | |
| Git でバージョン管理・差分確認できる | 記事の変更履歴も `git log` で追える |
| 外部依存なしで完全に自己完結 | |
| 記事 push で自動デプロイされる | GitHub 連携により手動デプロイ不要 |

#### 案 C: ヘッドレス CMS（Contentful / microCMS / Sanity）

```
CMS 管理画面 → CMS API → Next.js（ISR）
```

| 利点 | 欠点 |
|------|------|
| 専用の管理 UI で快適に執筆できる | 外部サービスへの依存が発生 |
| 画像管理・メディア管理が簡単 | 無料プランに制限がある場合がある |
| ISR で更新をリアルタイムに反映できる | セットアップコストがやや高い |

> **推奨**: 個人ブログかつコード品質重視であれば **案 B（Markdown + Git）** が最もシンプルで保守しやすい。
> 執筆体験を優先するなら **案 A（Notion API）** または **案 C（ヘッドレス CMS）** が適している。

## 5. 画面構成

```
/                    → トップページ（記事一覧）
/posts/[slug]        → 記事詳細
/tags/[tag]          → タグ別記事一覧
/categories/[cat]    → カテゴリ別記事一覧
/search?q=           → 検索結果
/about               → プロフィール・ブログについて（オプション）
```

---

## 6. データモデル（記事）

```typescript
type Post = {
  slug: string;          // URL 用の一意識別子
  title: string;         // 記事タイトル
  summary: string;       // サマリ（一覧カード・OGP 用）
  body: string;          // 本文（Markdown or リッチテキスト）
  tags: string[];        // タグ一覧
  category: string;      // カテゴリ
  publishedAt: string;   // 公開日（ISO 8601）
  updatedAt?: string;    // 更新日（オプション）
  published: boolean;    // 公開フラグ
};
```
