> 作成日: 2026-03-18
> 最終更新: 2026-03-18

---

## 1. プロジェクト概要

### 1.1 目的

技術系の記事を自由に記録・公開するための個人ブログアプリを構築する。
Zenn などのプラットフォームのガイドラインに縛られず、自分のペースでアウトプットを蓄積できる場所を持つことを目的とする。

### 1.2 背景

- 現在 Notion データベース + Notion API を使った自作ブログアプリが存在するが、コード品質・デザイン・機能に課題がある
- ゼロから再設計し、保守性・使いやすさ・見た目を向上させる
- Notion の記事は移行スクリプトにより Markdown ファイルへ一括移行済み

### 1.3 ターゲットユーザー

- **主な利用者**: 筆者本人（記事閲覧・管理）
- **副次的な利用者**: ブログを読みに来る不特定多数の読者

## 2. 機能要件

### 2.1 必須機能（Must）

| # | 機能 | 説明 |
|---|------|------|
| F-01 | 記事一覧表示 | 公開済み記事をカード形式で一覧表示する |
| F-02 | 記事詳細表示 | 記事本文（Markdown）を整形して表示する |
| F-03 | タグ・カテゴリ分類 | 記事にタグとカテゴリを付与し、絞り込み表示できる |
| F-04 | 記事検索 | キーワードで記事タイトル・サマリ・タグ・カテゴリを検索できる。ビルド時に生成した `public/search-index.json` をクライアントがフェッチして検索する |
| F-05 | コードシンタックスハイライト | コードブロックを言語ごとにカラーリングして表示する |
| F-06 | 閲覧数カウント | 記事詳細ページに閲覧数を表示する（Upstash Redis 使用） |

> **F-04 補足**: 本文全文検索は対象外。検索インデックスはビルド時に `public/search-index.json` として生成し CDN 配信する。HTML への全件埋め込みを避けることで初期ロードを軽量に保つ。
>
> **F-06 補足**: 閲覧数はベストエフォート方式。`@upstash/redis`（HTTP REST）を採用しているため Serverless Functions とのコネクション問題なし。ボット除外は User-Agent ベースで実施。同一ブラウザからの重複カウントは localStorage で防止。数値はあくまで参考値として扱う。

### 2.2 推奨機能（Should）

| # | 機能 | 説明 |
|---|------|------|
| F-07 | OGP 対応 | SNS シェア時にタイトル・説明・OGP 画像を表示する |
| F-08 | SEO メタデータ | 各記事に `<title>` / `<meta description>` を設定する |
| F-09 | Load More（段階的表示） | 記事一覧に「もっと見る」ボタンを設置し、10件ずつ追加表示する。初期表示 10件・追加表示 10件。クライアントサイドの state 管理で実装済み |
| F-10 | 目次（TOC）自動生成 | 長い記事の見出しから目次を自動生成して表示する |

### 2.3 将来対応（Could）

| # | 機能 | 説明 |
|---|------|------|
| F-11 | RSS フィード | `/feed.xml` で記事をフィード配信する |
| F-12 | ダークモード | ユーザーのシステム設定に応じて配色を切り替える（手動切り替えも可）|
| F-13 | 関連記事表示 | タグ・カテゴリをもとに関連記事をサジェストする |

## 3. 非機能要件

| カテゴリ | 要件 |
|----------|------|
| パフォーマンス | Lighthouse Score 90 以上（Performance / SEO / Accessibility） |
| レスポンシブ | スマートフォン・タブレット・PC すべてで正常に表示される |
| アクセシビリティ | 見出し構造・alt テキスト・キーボード操作の基本対応 |
| コード品質 | TypeScript 必須・ESLint / Prettier 設定・コンポーネント単位で責務を分離 |
| セキュリティ | API キーは環境変数で管理し、クライアントに露出しない |

## 4. 技術スタック

### 4.1 確定事項

| レイヤー | 技術 | 理由 |
|----------|------|------|
| フロントエンド | **Next.js 15 (App Router)** | SSG/ISR でブログに最適・TypeScript との相性◎ |
| 言語 | **TypeScript** | 型安全でコード品質を担保 |
| スタイリング | **Tailwind CSS** | ユーティリティファーストで素早く高品質な UI を構築 |
| シンタックスハイライト | **Shiki (rehype-pretty-code)** | コードブロックの見栄えを向上 |
| デプロイ | **Vercel** | Next.js との親和性が最も高い・GitHub 連携で自動デプロイ |
| ビルド方式 | **SSG（静的生成）** | 記事追加のたびにフルビルド。300件規模でも数分以内であれば許容（公開速度は優先しない）。`published: false` の下書きはビルド対象から除外される |
| 閲覧数 DB | **Upstash for Redis（Vercel Marketplace）** | Vercel KV が 2024年12月に廃止されたため代替として採用。HTTP REST API ベースの `@upstash/redis` を使用するため TCP コネクション問題なし。`HIKI_LOG_KV_REST_API_URL` / `HIKI_LOG_KV_REST_API_TOKEN` 環境変数で接続 |

### 4.2 コンテンツ管理: 案 B（Markdown + Git）✅ 採用

```
.md ファイル → git push → GitHub → Vercel 自動デプロイ
```

| 利点 | 備考 |
|------|------|
| シンプルでコード品質を保ちやすい | |
| Git でバージョン管理・差分確認できる | 記事の変更履歴も `git log` で追える |
| 記事 push で自動デプロイされる | GitHub 連携により手動デプロイ不要 |

## 5. 画面構成

```
/                    → トップページ（記事一覧）
/posts/[slug]        → 記事詳細
/tags/[tag]          → タグ別記事一覧
/categories/[cat]    → カテゴリ別記事一覧
/search?q=           → 検索結果
/about               → プロフィール・ブログについて
```

---

## 6. データモデル（記事）

```typescript
type Post = {
  slug: string;          // URL 用の一意識別子
  title: string;         // 記事タイトル
  summary: string;       // サマリ（一覧カード・OGP 用）
  body: string;          // 本文（Markdown）
  tags: string[];        // タグ一覧
  category: string;      // カテゴリ
  publishedAt: string;   // 公開日（ISO 8601）
  updatedAt?: string;    // 更新日（オプション）
  published: boolean;    // 公開フラグ
};
```

## 7. 環境変数

| 変数名 | 用途 | 備考 |
|--------|------|------|
| `NEXT_PUBLIC_SITE_URL` | OGP・sitemap の絶対 URL 生成 | ローカル: `http://localhost:3000` |
| `HIKI_LOG_KV_REST_API_URL` | 閲覧数カウント用 Upstash Redis REST エンドポイント | Vercel Storage 連携で自動設定（プレフィックス `HIKI_LOG_` はプロジェクト名）|
| `HIKI_LOG_KV_REST_API_TOKEN` | 閲覧数カウント用 Upstash Redis 認証トークン | Vercel Storage 連携で自動設定 |

## 9. 既知の課題・今後の対応候補

| 項目 | 内容 |
|------|------|
| 本文全文検索 | 現在はタイトル・サマリ・タグ・カテゴリのみ対象。将来的に Fuse.js 等でビルド時インデックスを生成する方法も検討可 |
| 画像管理 | Notion から移行した記事の画像は `public/images/` に手動配置。画像の alt テキストが空のものがある |
